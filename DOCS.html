StudyRecommendationAPI

Módulo backend independiente para una app mobile de gestión de tiempo de estudio universitario.

¿Qué hace este servicio?

  1. Recibe un syllabus (plan de estudios) y devuelve los temas extraídos de forma estructurada.
  2. Dado un tema, recomienda recursos educativos (videos, artículos, libros) ordenados por rating colaborativo.
  3. Registra feedback de estudiantes ("¿te sirvió?") y actualiza el rating del recurso en tiempo real.
Contexto de integración: la app mobile no sabe ni le importa qué pasa del lado del servidor. Solo consume endpoints REST y procesa JSONs. Sin autenticación.
Estado actual — todo es mock: no hay integración real con YouTube API ni procesamiento real de PDFs con IA. El endpoint de syllabus simula la extracción devolviendo temas hardcodeados según el nombre de la materia.

Reglas de código obligatorias

  • Sin var en ningún lugar. Siempre tipos explícitos: string, int, List<T>, AppDbContext, etc.
  • Entidades (Models/) nunca se exponen directamente. Siempre se usan DTOs.
  • Mensajes de respuesta al usuario en español. Código (clases, métodos, propiedades) en inglés.
  • Errores con Results.Problem(..., statusCode: N) — ProblemDetails estándar.
  • No hay autenticación ni autorización en ningún endpoint.
  • Cada grupo de endpoints vive en su propio archivo en Extensions/ como método de extensión sobre WebApplication.

Endpoints

POST /api/syllabus/process Procesa un syllabus y devuelve temas estructurados

Request body

CampoTipoDescripción
subjectNamestringREQNombre de la materia. Clave de match para detectar duplicados.
careerstringoptCarrera. Solo se usa al crear.
institutionstringoptInstitución. Solo se usa al crear.
fileBase64stringoptSe recibe pero se ignora (mock).

Lógica

  • Busca materia por Name exacto (case-sensitive).
  • Si existe → devuelve los temas existentes sin crear nada.
  • Si no existe → crea Subject + Topics mockeados desde SeedData.GetMockTopics().
  • Si el nombre no tiene mock definido → genera temas genéricos (3 unidades, 2 temas cada una).

GET /api/recommendations/{topicId} Recursos de un tema ordenados por rating descendente

Path param

ParamTipoDescripción
topicIdintID del tema obtenido de syllabus/process o subjects/{id}/topics

Ordenamiento

  • Primero los recursos con al menos 1 voto.
  • Entre ellos, por rating descendente (positiveVotes / totalVotes).
  • Empate de rating → mayor totalVotes primero.
  • Recursos sin votos van al final (rating = 0).

      "totalVotes": 60
    }
  ]
}

404 topicId no existe.
POST /api/feedback Registra voto de un estudiante sobre un recurso

Request body

CampoTipoDescripción
resourceIdintREQID del recurso a votar
userIdstringREQIdentificador del usuario (string libre, ej: "user001")
isHelpfulboolREQtrue = útil, false = no útil

Lógica

  • Valida que userId no sea vacío.
  • Verifica que el recurso exista.
  • Verifica que el usuario no haya votado ya ese recurso.
  • Crea registro en Feedbacks y actualiza PositiveVotes o NegativeVotes en Resources.

Response 200

{
  "message": "Feedback registrado",
  "resourceId": 1,
  "newRating": 0.88,
  "totalVotes": 52
}
200 Feedback guardado y contadores actualizados.
400 userId vacío.
404 resourceId no existe.
409 El usuario ya votó ese recurso.

Distribución de votos en los recursos

  • Alta aprobación: 45–60 positivos, 2–5 negativos → rating ~0.90–0.96
  • Aprobación media: 12–30 positivos, 3–8 negativos → rating ~0.60–0.88
  • Pocos votos: 3–5 positivos, 1–2 negativos → rating ~0.70–0.80
  • Sin votos: 0/0 → rating 0, aparecen al final de las recomendaciones

También se generan registros en Feedbacks para los primeros 30 recursos (con userId del estilo user001..user010), coherentes con los votos cargados.

Cálculo de rating

int total = resource.PositiveVotes + resource.NegativeVotes;
double rating = total == 0 ? 0 : Math.Round((double)resource.PositiveVotes / total, 2);

Rango: 0.00 a 1.00. Recursos sin votos devuelven 0 y se muestran al final. No hay Bayesian average ni piso mínimo de votos en esta etapa.

Startup (Program.cs)

// 1. Registra EF Core con SQLite
builder.Services.AddDbContext<AppDbContext>(options =>
    options.UseSqlite(connectionString));

// 2. Registra CORS permisivo
builder.Services.AddCors(o => o.AddDefaultPolicy(
    p => p.AllowAnyOrigin().AllowAnyHeader().AllowAnyMethod()));

// 3. Al arrancar: crea DB si no existe y seedea si está vacía
using (IServiceScope scope = app.Services.CreateScope()) {
    AppDbContext db = scope.ServiceProvider.GetRequiredService<AppDbContext>();
    db.Database.EnsureCreated();
    await SeedData.SeedAsync(db);
}

// 4. Mapea todos los endpoints
app.MapSyllabusEndpoints();
app.MapRecommendationEndpoints();
app.MapFeedbackEndpoints();
app.MapSubjectEndpoints();
La DB se crea en bin/Debug/net8.0/studyrecommendation.db. Si querés resetear el seed, borrá ese archivo y reiniciá.

Pendiente / siguiente etapa

Lo que falta para pasar de mock a real

  • Procesamiento real de PDF: reemplazar la lógica de SyllabusEndpoints que ignora fileBase64 por extracción real de texto (ej: Azure Document Intelligence, iTextSharp, o llamada a modelo de lenguaje).
  • Recursos reales: integrar YouTube Data API y/o scrapers de sitios educativos para poblar Resources con datos reales en lugar de seed hardcodeado.
  • Autenticación: actualmente UserId es un string libre. Si la app mobile agrega auth (JWT, etc.), el userId debería venir del token.
  • Migraciones EF Core: cuando el schema cambie en producción, reemplazar EnsureCreated() por migraciones reales (dotnet ef migrations add).
  • Rating mejorado: considerar Bayesian average o Wilson score para recursos con pocos votos.
  • Paginación: los endpoints de recursos y materias no tienen paginación. Podría ser necesario con muchos datos.

Decisiones de diseño a tener en cuenta

  • El match de materia en /api/syllabus/process es case-sensitive y por nombre exacto. Si la app envía "ingeniería de datos II" (minúscula), no va a hacer match. Considerar normalizar o usar EF.Functions.Like.
  • SeedData.GetMockTopics() es el "catálogo" de materias conocidas. Para agregar una nueva materia con mock, se agrega una entrada a ese dictionary.
  • Los votos en Resources son contadores denormalizados (no se calculan sumando Feedbacks). Son la fuente de verdad para el rating. El seed los carga directamente.
  • Los feedbacks del seed son parciales (solo los primeros 30 recursos, máximo 10 feedbacks cada uno) — son ilustrativos, no cubren todos los votos.
